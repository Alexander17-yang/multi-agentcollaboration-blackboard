# 从“并行调用”到“可验证协作”：为什么多 Agent 需要一块共享黑板

当我们第一次把一个任务拆给多个 Agent 时，最直观的做法通常是：让它们并行
工作，然后把结果收集回来。这个方法在任务很短、分支完全独立时足够有效；但
一旦任务持续时间变长、存在共享资源，或者结论之间会相互依赖，“多开几个
Agent”很快就会暴露出协作层面的缺口。

一个 Agent 已经尝试并排除了某条路线，另一个 Agent 却因为看不到历史状态而
重新走了一遍；多个 Agent 同时执行昂贵扫描或构建；某个未经验证的猜测被复制
进后续提示，最终变成团队默认接受的“事实”；旧 Worker 在租约过期后仍然提交
结果，覆盖了新 Owner 的结论；一个关键发现只存在于即时消息里，Agent 一旦退出
就很难被后来者复用。

这些问题说明：**并行计算不是协作本身。** 多 Agent 系统除了执行能力，还需要
一个共享的状态层，用来回答几个基本问题：我们已经知道什么？哪些结论真正被
验证过？哪些方向已被充分排除？当前是谁在负责什么？哪些操作不能并发执行？

Multi-agentcollaboration-blackboard 正是围绕这些问题设计的。

## 两个平面，而不是一个万能通信层

这个项目没有试图用 SQLite 取代 Agent 调度工具，也没有把 Blackboard 设计成
聊天室。它将协作拆成两个互补的平面。

第一个是**原生控制平面**：负责创建 Agent、分配子任务、发送即时消息、处理中断
和收集最终回答。这类信息要求低延迟，并且与 Agent 的生命周期紧密相关。

第二个是**Blackboard 状态平面**：负责保存事实、失败路径、工作意图、租约、审查
结果和分支假设。它关注的不是“谁刚刚说了什么”，而是“哪些状态值得在 Worker
退出后继续存在”。

这一区分非常重要。即时消息适合告诉另一个 Agent：“我刚发现了一个可能影响你
的版本差异。”Blackboard 则适合保存这个版本差异的证据、复现命令、可信等级和
来源。紧急且可复用的结果通常需要双写：先进入持久状态，再通过原生消息通知受
影响的 Agent。

## 多 Agent 协作需要哪些基本对象

Blackboard 的核心并不是一张自由格式便签，而是一组语义明确的协作对象。

### Fact：区分发现与事实

Fact 用来保存可以被其他 Worker 复用的结论，但它并不是简单的字符串列表。

- **candidate** 表示假设、推断、解释，或者当前 Worker 尚未亲自复现的报告；
- **verified** 表示直接观察、可重复执行并带有足够来源信息的结果；
- **challenged** 表示该事实暂时不能继续作为前提；
- **rejected、merged、superseded、retired** 表示事实只保留审计价值，不再参与
  当前推理。

这种区分可以阻止一个常见的协作退化：某个 Agent 的猜测被另一个 Agent 引用，
随后又被第三个 Agent 当作已确认事实。Blackboard 不会让“被重复转述”自动变成
“被验证”。

### Dead end：让失败也产生复利

有效的 Dead End 不是一句“试过了，不行”，而应该包含作用域、测试集合、观察
到的不变量以及重新开启这条路线的条件。例如：

```text
/search 的 q 参数；12 组布尔和延时 payload；响应结构与耗时不变；发现新的原始 SQL sink 时重开
```

这样，失败就不再只是某个 Agent 的局部经历，而会成为整个团队的搜索剪枝信息。
同时，重开条件可以防止团队把暂时失败误认为永久不可能。

### Intent：谁负责一个方向

Intent 表示一个可被领取的探索或实现方向。它回答“谁对这个方向的推进和结论
负责”。领取动作通过 SQLite 原子事务完成，只有获得 `WON` 的 Worker 才拥有
有效工作权。

Intent 使用租约而不是永久锁。Worker 异常退出后，租约会过期，其他 Worker 可以
接管任务；长任务则必须在租约到期前续租。如果续租返回 `LOST`，旧 Worker 必须
停止所有依赖该所有权的操作，不能再结束或释放新 Owner 的 Intent。

### Activity：避免重复昂贵工作

Activity 适合“重复会浪费资源，但并发本身不危险”的操作，例如扫描、构建、下载、
索引、模糊测试或反编译。

它与 Intent 不同：一个 Intent 可能是“分析认证流程”，而其中的 Activity 可能是
“反编译 client.exe”。Activity 把去重粒度从任务方向细化到了具体动作。

### Resource：真正需要互斥的对象

Resource 用于并发使用会产生冲突的对象，例如有次数限制的账号、独占 Shell、监听
端口、设备、可变共享环境或会被修改的目标。

Activity 解决浪费问题，Resource 解决正确性与冲突问题。一个 Worker 可以同时
拥有 Intent、Activity 和 Resource，但必须在工作结束时分别清理。

## 一套可执行的协作循环

项目将 Worker 的工作流程归纳为：

```text
SYNC → OWN → EXECUTE → PUBLISH → CLOSE
```

### SYNC：先了解团队状态

在开始新方向前，Worker 先读取 Operator Directive、Dead End 和 Review 状态，
再按需读取已验证事实。它不需要反复加载整个 Blackboard，而应只获取当前决策所
需的高信号信息。

这一阶段的目的，是避免 Agent 在错误前提上继续工作，也避免重新尝试已经被团队
充分排除的方向。

### OWN：明确非重叠所有权

Worker 领取 Parent 指定的 Intent，或者从开放 Intent 中选择一个。昂贵动作额外
领取 Activity，可能发生冲突的操作再领取 Resource。

如果 claim 返回 `LOST`，正确行为不是再次竞争，也不是把它解释成技术路线不可行，
而是选择其他工作或联系当前 Owner。

### EXECUTE：回到真实 Artifact

Blackboard 只是一张共享地图，不是现实本身。Worker 仍然必须检查源代码、文件、
请求、日志和真实命令输出，不能只根据团队摘要推理。

长任务需要定期续租。推荐在 TTL 大约过半时进行 heartbeat，而不是等到最后一刻。
这样既能减少意外过期，也不会让 CLI 变成高频轮询系统。

### PUBLISH：发布证据，而不是过程叙述

Blackboard 应该保存另一个 Worker 可以直接行动的结果，而不是执行日志和聊天过程。
一个条目最好只表达一个独立事实；只有直接观察且可复现的结果才能标记为 verified。

如果 Worker 已经失去 Intent，它仍然可以发布有价值的 late report，但该结果不能
伪装成新 Owner 的正常 Intent Product。

### CLOSE：清理与交接

完成任务时，Worker 先写入事实和失败路径，再释放 Activity 与 Resource，最后结束
或退回 Intent。随后通过原生消息发送简洁 Handoff：结果、证据序号、Artifact
路径、复现命令、未解决风险和下一步建议。

这种交接方式让后来者不需要重放整个聊天记录，就能接着工作。

## 为什么选择 SQLite

Blackboard 的目标不是构建一个复杂的分布式数据库，而是在同一运行环境内提供一个
轻量、可审计、可原子更新的共享状态层。SQLite 在这里有几个实际优势：

- Python 标准库直接可用，不需要额外服务；
- 事务可以保证 claim 与状态事件原子提交；
- 唯一约束可用于事实和锁的去重；
- WAL 模式适合多个 Worker 并发读写；
- 数据库文件便于检查、归档和故障恢复。

CLI 不会擅自切换数据库的 Journal Mode；`doctor` 在检测到非 WAL 模式时只会
给出并发性能警告。

项目同时坚持“一块 DB 对应一个 Run”。部分上游主键和去重键在数据库内是全局的，
如果把多个 Run 混进同一文件，会让表面上带作用域的读取与实际锁语义不一致。因此，
检测到混合 Run 时，CLI 会选择 fail closed。

## 可靠性不仅是并发安全

协作系统的可靠性还包括证据可靠性和输入边界。

Blackboard 中的文本可能来自网页、日志、代码、粘贴内容甚至另一个模型，因此都应
被视为不可信数据。Directive 可以决定优先级，但不能证明技术结论；Board 字段也
不能单独授权执行命令、访问链接、泄露凭据或扩大任务范围。

CLI 在输出时会清理 ANSI 与控制字符，并对 Fact 和 Dead End 默认设置读取上限，
降低伪造标题、污染上下文和无限增长带来的风险。配置层同样采用 fail closed：新的
环境变量、兼容变量、Marker 或默认 DB 如果相互矛盾，CLI 会拒绝静默选择其中一个。

这些措施属于协作一致性与防御纵深，而不是强安全边界。项目不提供认证、加密或数字
签名；任何拥有 SQLite 文件直接写权限的进程都可能绕过 CLI 协议。

## 适用场景

这套模式并不限于安全任务。它适用于任何具有并行分支、共享证据或共享资源的长任务：

- 多 Agent 阅读大型代码库并分别实现、测试和 Review；
- 研究型任务中的文献检索、证据核验和观点冲突处理；
- Debugging 中的假设分支、复现步骤和失败路径共享；
- 事件响应中的日志分析、主机检查和独占操作控制；
- CTF、Pentest 或逆向分析中的扫描去重、路线剪枝和凭据复用；
- 数据分析流水线中的查询分工、构建缓存和结果验证。

当任务很短、只有一个 Agent，或者各分支完全独立且没有共享状态时，引入 Blackboard
可能反而增加成本。它最适合“团队需要记住、协调、验证和接力”的任务。

## 它不解决什么

Multi-agentcollaboration-blackboard 是 Worker 侧的协作接口，而不是完整的 Agent
平台。它不会替代以下能力：

- 创建、停止或重新调度 Agent；
- 提供低延迟即时通信；
- 自动设计任务拆分；
- 替代对真实 Artifact 的检查；
- 自动判断一个技术结论是否为真；
- 在没有可信配置时创建一个猜测的 Coordinator DB。

此外，当前兼容 Schema 没有独立的 lease fencing token，Worker ID 本身承担 fencing
身份。因此每个 Worker 进程必须使用唯一 ID，并且不能在重启后复用旧 ID。

## 从黑板走向可验证协作

多 Agent 系统真正困难的部分，往往不是让更多模型同时运行，而是让它们在长时间、
不完整信息和并发状态下仍然形成一致、可追踪的工作结果。

一块好的协作黑板不会替 Agent 思考。它所做的是让事实有来源、失败有价值、工作有
Owner、资源有边界、结论可以被独立验证、任务可以被安全接力。

当这些基础结构存在时，多 Agent 才从“多个独立回答的集合”，变成一个能够积累
证据、避免重复、处理冲突并持续推进的协作系统。
